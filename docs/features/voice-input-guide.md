# 语音输入前端说明

本文档说明 SmartDictate 当前语音输入能力的实现方式、使用限制和后续扩展方向。

## 当前实现方式

前端通过浏览器 Web Speech API 完成语音识别，识别结果会先写入“实时转写”文本框。用户可以手动编辑识别结果，再点击“整理文本”调用后端接口完成口癖词清理、热词纠错和标点整理。

Web Speech API 是浏览器提供的语音识别能力，不是本项目自研的小模型，也不是当前后端调用的在线语音识别 API。实际识别能力由浏览器实现决定，Chrome 和 Edge 通常会依赖浏览器厂商的语音服务，因此可能需要网络和麦克风权限。

当前前端代码按职责拆分为：

```text
frontend/src/main.js
frontend/src/modules/api-client.js
frontend/src/modules/speech-recognition.js
frontend/src/modules/html.js
frontend/src/styles.css
```

各文件职责：

- `main.js`：页面状态、事件绑定和 UI 渲染。
- `api-client.js`：封装后端 API 请求。
- `speech-recognition.js`：封装浏览器语音识别能力。
- `html.js`：封装 HTML 转义等通用工具。
- `styles.css`：页面布局和视觉样式。

## 使用要求

- 推荐浏览器：最新版 Chrome 或 Edge。
- 推荐访问地址：`http://127.0.0.1:5173`。
- 首次点击“开始语音输入”时，需要允许浏览器麦克风权限。
- 如果浏览器不支持 Web Speech API，页面会禁用语音输入按钮并提示原因。

在图书馆等不方便出声的环境中，可以先直接在“实时转写”文本框粘贴一段文字，再点击“整理文本”验证前后端联动。语音识别本身需要实际声音输入才能验证。

## 前后端联动流程

1. 用户点击“开始语音输入”。
2. 浏览器把语音实时识别为文本。
3. 前端把最终识别结果写入“实时转写”文本框。
4. 用户点击“整理文本”。
5. 前端请求后端 `/api/transcripts/process`。
6. 后端返回整理结果和指标信息。
7. 前端展示整理结果，并刷新最近记录。

## 工程取舍

当前方案优点：

- 不需要个人服务器。
- 不需要申请第三方 API 密钥。
- 响应速度快，适合作为 MVP 和演示闭环。
- 成本低，便于评委本地复现。

当前方案限制：

- 识别质量依赖浏览器和系统语音服务。
- 不同浏览器支持程度不同。
- 对长音频、嘈杂环境和专业词汇的准确度有限。

后续可以把在线语音识别 API 作为可选增强通道，而不是替换当前默认能力。这样即使没有 API 密钥，项目仍然可以运行。
