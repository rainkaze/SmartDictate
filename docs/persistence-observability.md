# 持久化与可观测性说明

## 为什么从 JSON 文件升级到 SQLite

早期版本使用 JSON 文件保存历史记录和自定义热词，优点是实现简单、便于快速验证 MVP。但是 JSON 文件不适合继续承载更多真实业务行为：

- 并发读写能力弱，多个请求同时写入时容易产生覆盖风险。
- 查询能力有限，只能整体读写文件，不适合后续筛选、分页和统计。
- 数据结构缺少约束，长期维护时容易出现脏数据。
- 很难自然表达索引、主键、唯一性等基础数据建模能力。

当前版本改为使用 SQLite 作为本地持久化存储。SQLite 不需要单独安装数据库服务，仍然符合本项目“本地优先、低成本、易复现”的定位，同时具备更接近真实工程的数据管理能力。

## 当前数据表

### transcripts

保存语音转写和文本整理后的历史记录。

```text
id              文本记录主键
raw_text        原始识别文本
processed_text  整理后的文本
scene           输入场景
metrics_json    文本处理指标
created_at      创建时间
```

### hotwords

保存用户自定义热词。内置热词仍然来自 `backend/app/config/text_rules.json`，用户热词保存在 SQLite 中。

```text
source      识别词
target      标准词
created_at  创建时间
```

## 数据访问层设计

项目没有直接在接口函数里写 SQL，而是通过服务类隔离数据访问：

```text
backend/app/services/storage.py   历史记录持久化
backend/app/services/hotwords.py  自定义热词持久化
```

这样做的好处是：

- API 层只关心请求和响应，不关心数据如何保存。
- 后续如果从 SQLite 切换到 MySQL 或 PostgreSQL，主要修改服务层即可。
- 单元测试可以直接覆盖数据访问行为，避免只做页面级验证。

## 中间件设计

项目新增了 `RequestContextMiddleware`，负责为每个请求补充：

- `X-Request-ID`：请求追踪 ID，便于定位一次前后端交互。
- `X-Process-Time-Ms`：后端处理耗时，便于观察接口响应速度。
- 基础访问日志：记录请求方法、路径、状态码和耗时。

这属于项目的可观测性基础能力。它不会显著增加开发成本，但能体现后端服务在排查问题、观察性能方面的工程意识。

## 健康检查增强

`GET /api/health` 不再只返回简单的 `ok`，还会返回：

```text
服务版本
存储引擎
数据库是否可用
历史记录数量
自定义热词数量
```

这使得前端和评委可以快速判断后端是否真正可用，而不只是进程是否启动。

## 工程取舍

本项目没有直接引入 MySQL、Redis 或消息队列，因为它们会显著提高本地复现成本，也不符合 72 小时作品的交付边界。SQLite 是更合适的中间方案：

- 比 JSON 文件更规范。
- 比外部数据库更容易启动和演示。
- 足够支撑当前历史记录、热词字典和后续统计能力。
- 保留了向企业级数据库迁移的清晰路径。
