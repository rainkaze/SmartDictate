# 开发计划

本文档用于记录 SmartDictate 在实训周期内的开发顺序、交付目标和后续扩展方向。每个阶段应尽量通过独立 PR 交付，保证提交记录清晰、可追踪、可评审。

## 第 1 阶段：项目基础框架

目标：完成一个可以本地启动、前后端职责清晰的最小工程骨架。

- 初始化仓库基础文件：README、.gitignore、requirements、pyproject、EditorConfig。
- 搭建 FastAPI 后端，提供健康检查、文本整理和历史记录接口。
- 搭建 Vite 前端，完成基础页面布局和接口调用入口。
- 配置 GitHub Actions 同步到 Gitee，保证提交后可自动镜像。
- 补充中文启动说明和后端接口说明文档。

验收标准：

- 后端可以通过 `uvicorn backend.app.main:app --reload --host 127.0.0.1 --port 8000` 启动。
- 前端可以通过 `npm run dev` 启动。
- 访问 `http://127.0.0.1:5173` 能看到 SmartDictate 页面。
- 访问 `http://127.0.0.1:8000/docs` 能看到后端接口文档。

## 第 2 阶段：语音输入闭环

目标：让用户可以完成从语音输入到文本整理的一条完整路径。

- 前端接入浏览器 Web Speech API。
- 支持开始录音、停止录音、实时识别状态展示。
- 支持手动编辑识别文本。
- 前端调用后端 `/api/transcripts/process` 完成文本整理。
- 展示字数、清理口癖词数量、耗时等基础指标。

验收标准：

- 用户可以通过浏览器麦克风完成语音转文字。
- 点击“整理文本”后，前端能展示后端返回的整理结果。
- 后端不可用时，前端有明确错误提示。

## 第 3 阶段：文本增强能力

目标：提升语音输入结果的可用性，让作品体现产品思考。

- 完善口癖词清理规则。
- 增加热词纠错能力，例如技术词、课程名、产品名、人名。
- 增加场景模式：通用输入、会议纪要、学习笔记、聊天回复、代码注释。
- 为文本处理逻辑补充 pytest 单元测试。

验收标准：

- 常见误识别词可以被替换成标准词。
- 不同场景能生成不同格式的输出。
- 文本处理核心逻辑有自动化测试覆盖。

## 第 4 阶段：历史记录和体验优化

目标：让产品更接近日常工具，而不是一次性 Demo。

- 保存最近整理记录到本地 JSON 文件。
- 前端展示最近记录。
- 支持从历史记录回填结果。
- 优化移动端布局和异常状态。

验收标准：

- 用户整理文本后，刷新历史记录可以看到新结果。
- 点击历史记录可以把内容重新填入编辑区。
- 页面在常见笔记本屏幕和手机宽度下布局正常。

## 第 5 阶段：在线语音识别 API 扩展

目标：在保留低成本默认方案的基础上，提供更高准确率的可选识别通道。

- 设计语音识别服务抽象层。
- 保留 Web Speech API 作为默认识别方案。
- 后端预留在线语音识别 API 的配置入口。
- API 密钥通过环境变量读取，禁止写入代码和 README。
- 在 README 中说明不同识别方案的成本、准确度、响应速度和隐私权衡。

验收标准：

- 即使未配置在线 API，项目仍然可以正常运行。
- 配置在线 API 后，可以通过独立接口或模式切换使用增强识别。
- 敏感密钥不会出现在 Git 提交记录中。

## 第 6 阶段：交付材料

目标：完成评审需要的说明、演示和复现材料。

- 完善 README：项目介绍、功能说明、启动方式、架构说明、技术取舍。
- 补充截图或 GIF。
- 录制 Demo 视频，并把可访问链接放在 README 显眼位置。
- 检查 PR 描述、commit 分布和依赖说明。
- 确认 GitHub 和 Gitee 仓库均可访问。

验收标准：

- 只看 README 也能理解项目价值和启动方式。
- Demo 视频覆盖语音输入、文本整理、复制、历史记录等核心流程。
- 主分支代码在本地可以复现演示效果。
